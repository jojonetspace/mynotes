# 商品复制功能详细分析

## 一、功能概述

商品复制功能允许将一个源商品复制到多个目标店铺,支持同平台、跨平台、跨类目的商品复制。系统会自动进行属性映射、数据转换和验证,尽可能保证复制后的商品可以直接发布。

### 核心特性
- 并行处理多个目标店铺,提升性能
- 智能属性映射(数据库配置 → 字符串匹配 → 向量相似度)
- 向量预计算,避免重复调用Embedding API
- 自动验证商品完整性
- 智能状态继承(源商品已完善 → 目标商品也尽量保持已完善)

---

## 二、核心流程图

```
用户选择源商品 + 目标店铺列表
    ↓
ShopItemService.CopyShopItem() / CopyShopItemBySiIdAsync()
    ↓
【Phase 1】源商品预处理
  - 获取源商品店铺和平台信息
  - 执行平台特定的预处理逻辑
    ↓
【Phase 2】源商品向量预计算 (性能优化)
  - 收集所有属性名称和属性值
  - 批量调用Embedding API生成向量
  - 缓存向量供后续使用
    ↓
【Phase 3】并行处理多个目标店铺

    ├─ 步骤1: CopyNew() - 创建新商品对象

    ├─ 步骤6: ShopItemValidation() - 商品验证
    └─ 步骤7: 保存到数据库
  }
    ↓
返回复制后的商品列表
```

## 三、复制前置校验

### 3.1 店铺授权状态检查

在执行商品复制之前，系统会对目标店铺列表进行授权状态过滤，只有授权状态为 `Authorized` 的店铺才会被纳入复制目标。

**代码位置**: `ShopItemService.cs` Line 2653-2672（`CopyShopItemBySiIdAsync` 方法）

```csharp
public async Task<List<ShopItem>> CopyShopItemBySiIdAsync(long si_id, List<int> targetShopIds)
{
    // 过滤掉授权失效的店铺
    var validShopIds = new List<int>();
    foreach (var shopId in targetShopIds)
    {
        var shop = ShopService.GetShop(shopId);
        if (shop != null && shop.session_status == SessionStatus.Authorized.ToString())
        {
            validShopIds.Add(shopId);
        }
    }

    // 如果没有有效的目标店铺，直接返回空列表
    if (validShopIds.Count == 0)
    {
        return new List<ShopItem>();
    }
    
    // ... 后续复制逻辑使用 validShopIds
}
```

**授权状态过滤规则**:

| 店铺授权状态 | 是否允许复制 | 说明 |
|-------------|------------|------|
| Authorized（已授权） | ✅ 允许 | 正常复制 |
| Unauthorized（未授权） | ❌ 跳过 | 静默过滤，不报错 |
| Expired（已过期） | ❌ 跳过 | 静默过滤，不报错 |
| Invalid（失效） | ❌ 跳过 | 静默过滤，不报错 |

**注意事项**:
- 授权过滤仅在 `CopyShopItemBySiIdAsync`（异步版本）中实现
- `CopyShopItem`（同步版本，Line 2690）不包含授权检查，调用方需自行确保店铺授权有效
- 如果所有目标店铺都未授权，返回空列表，不抛异常
- 过滤是静默的，前端不会收到"某店铺因授权失效被跳过"的提示

### 步骤1: 创建新商品对象 (CopyNew)

**代码位置**: `ShopItemService.cs` Line 4221-4243

```csharp
private ShopItem CopyNew(ShopItem shopItem, ShopLC shop)
{
    // 1. 深拷贝源商品(使用JSON序列化/反序列化)
    var item = JsonConvert.DeserializeObject<ShopItem>(
        JsonConvert.SerializeObject(shopItem));
    
    // 2. 重置关键字段
    item.si_id = 0;                    // 新商品ID
    item.copy_from = shopItem.si_id;   // 记录复制来源
    item.plf_id = shop.plf_id;         // 目标平台
    item.country_id = shop.country_id; // 目标站点
    item.shop_id = shop.shop_id;       // 目标店铺
    
    // 3. 清空平台相关字段
    item.shop_i_id = string.Empty;     // 平台商品ID
    item.shop_g_i_id = string.Empty;   // 平台全球商品ID
    
    // 4. 重置SKU字段
    item.Skus.ForEach(s => {
        s.autoid = 0;
        s.si_id = 0;
        s.shop_i_id = string.Empty;
        s.shop_sku_id = string.Empty;
        s.shop_g_i_id = string.Empty;
        s.down_sku_id = string.Empty;
    });
    
    // 5. 设置操作状态 (关键!)
    item.operation_status = (
        shopItem.operation_status.Is(OperationStatus.Completed.ToString()) ||
        shopItem.operation_status.Is(OperationStatus.Uploaded.ToString()) ||
        shopItem.operation_status.Is(OperationStatus.Uploading.ToString())
    ) ? OperationStatus.Completed.ToString()    // 源商品已完善 → 目标也是已完善
      : OperationStatus.Incomplete.ToString();  // 源商品未完善 → 目标也是未完善
    
    // 6. 清空状态字段
    item.status = string.Empty;
    item.listing_status = string.Empty;
    item.operation_msg = string.Empty;
    
    return item;
}
```

**字段处理说明**:

| 字段类型 | 处理方式 | 说明 |
|---------|---------|------|
| 主键字段 | 重置为0 | si_id, autoid |
| 平台字段 | 更新为目标 | plf_id, country_id, shop_id |
| 平台ID字段 | 清空 | shop_i_id, shop_g_i_id, shop_sku_id |
| 状态字段 | 智能继承 | operation_status |
| 其他字段 | 保持不变 | title, pic, brand等 |

```csharp
private void CleanResourceUploadId(ShopItem shopitem)
{
    // 1. 清理商品图片的upload_id
    if (shopitem.media_info.img_list.Count > 0)
    {
        shopitem.media_info.img_list.ForEach(x => { 
            x.Remove("upload_id"); 
        });
    }
    
    // 2. 清理视频ID
    shopitem.media_info.video_id = string.Empty;
    
    // 3. 清理尺码表upload_id
    if (shopitem.media_info.size_chart != null)
    {
        shopitem.media_info.size_chart.upload_id = string.Empty;
    }
    
    // 4. 清理SKU图片的upload_id
    foreach (var sku in shopitem.Skus)
    {
        if (sku.ImgProperty != null)
        {
            sku.ImgProperty.upload_id = string.Empty;
        }
    }
    
    // 5. 重置物流信息(保留部分字段)
    var logistic_info = new LogisticInfo();
    logistic_info.cod = shopitem.logistic_info.cod;
    logistic_info.is_dangerous = shopitem.logistic_info.is_dangerous;
    logistic_info.item_dimension = shopitem.logistic_info.item_dimension;
    logistic_info.item_weight = shopitem.logistic_info.item_weight;
    shopitem.logistic_info = logistic_info;
    
    // 6. 清空产品属性(需要重新映射)
    shopitem.product_properties = new List<ProductProperty>();
    
    // 7. 只保留已选中的销售属性
    var sales_properties = shopitem.sales_properties
        .Where(x => x.is_checked == true).ToList();
    shopitem.sales_properties = sales_properties;
}
```

**为什么要清理upload_id?**
- upload_id是平台特定的资源标识
- 跨店铺时,资源需要重新上传到目标店铺
- 保留旧的upload_id会导致发布失败

## 五、复制后是否需要编辑才能发布?

### 答案: 取决于源商品状态和验证结果

### 情况1: 可以直接发布 ✅

**条件**:
1. 源商品的`operation_status`是以下之一:
   - `Completed` (已完善)
   - `Uploaded` (已发布)
   - `Uploading` (发布中)
2. 商品验证通过 (`ShopItemValidation`返回`valid=true`)
3. 属性映射成功(必填属性都有值)

**结果**:
- 复制后的商品`operation_status = Completed`
- 前端显示"已完善"图标(绿色对勾)
- 可以直接点击"发布"按钮发布到平台
- 不需要额外编辑

**适用场景**:
- 同平台同类目复制
- 跨店铺但平台和类目相同
- 属性映射配置完善的跨平台复制

---

### 情况2: 需要编辑后才能发布 ⚠️

**条件**:
1. 源商品的`operation_status`是`Incomplete`(待完善)
2. 或者商品验证失败(`valid=false`)
3. 或者属性映射不完整(必填属性缺失)

**结果**:
- 复制后的商品`operation_status = Incomplete`
- 前端显示"待完善"图标(红色感叹号)
- 必须先编辑完善商品信息,才能发布
- 点击"发布"按钮会提示"请先完善商品信息"

**常见需要编辑的场景**:
- 跨平台复制(Lazada → Shopee),属性无法完全映射
- 跨类目复制,目标类目的必填属性与源类目不同
- 源商品本身就不完整
- 目标平台有特殊要求(如Facebook的catalog_id)
- 属性映射配置不完善

**需要编辑的内容**:
- 补充缺失的必填属性
- 调整不符合目标平台要求的字段

### 8.1 同平台同类目复制

**特点**:
- 类目属性差异大

## 九、常见问题

### Q1: 为什么复制后商品状态是"待完善"?

**可能原因**:
1. 源商品本身就是`Incomplete`状态
2. 跨平台/跨类目,属性映射不完整
3. 目标平台的必填属性缺失
4. 商品验证失败(价格/库存/图片等不符合要求)

**解决方法**:
- 查看`operation_msg`字段,了解具体原因
- 编辑商品,补充缺失的信息
- 完善属性映射配置

### Q2: 复制后图片/视频丢失了?

**原因**:
- 跨店铺复制时,`upload_id`被清空
- 图片/视频需要重新上传到目标店铺

**解决方法**:
- 系统会自动处理,发布时会重新上传资源
- 如果是D+上传,会在后台异步处理
- 等待资源上传完成后再发布

### Q3: 如何提高复制后直接发布的成功率?

**方法**:
1. 完善属性映射配置(商家级别)
2. 确保源商品信息完整
3. 同平台同类目复制成功率最高
4. 使用系统推荐的类目映射
5. 定期更新平台规则配置

### Q4: 复制很慢怎么办?

**优化方法**:
1. 检查网络连接
2. 减少单次复制的店铺数量
3. 避免高峰期操作
4. 联系管理员检查服务器性能
5. 检查Embedding API是否正常

## 十二、属性映射方法体系详解

> **补充时间**: 2026-03-12  
> **目的**: 详细说明 `ShopItemCatMapService` 中 BuildKey 方法的设计意图和商品复制功能实际使用的映射策略

### 12.1 BuildKey 方法体系概述

`ShopItemCatMapService` 类提供了一套完整的 BuildKey 方法体系，用于生成跨平台商品数据映射的唯一键。这些键用于在数据库中存储和查询不同平台间的属性映射关系。

#### Key 的设计结构

所有 Key 都遵循统一格式：
```
{类型前缀}-{商家ID}-{源平台ID}-{目标平台ID}-{源数据ID}
```

返回值是一个元组：
- `hash_id`: Key 的哈希值，用于数据库索引和快速查询
- `mapping_key`: 完整的映射键字符串，用于精确匹配

#### Key 类型前缀

| 前缀 | 含义 | 示例 |
|------|------|------|
| `c` | Category (类目) | `c-1001-shopee-lazada-100001` |
| `ca` | Category Attribute (商品属性) | `ca-2-shopee-lazada-brand` |
| `cav` | Category Attribute Value (商品属性值) | `cav-2-shopee-lazada-nike` |
| `sa` | Sales Attribute (销售属性) | `sa-2-shopee-lazada-color` |
| `sav` | Sales Attribute Value (销售属性值) | `sav-2-shopee-lazada-red` |

### 12.2 两级映射机制

#### 商家级别映射 (co_id = 实际商家ID)
- 允许商家自定义映射规则
- 优先级高于系统级别
- 适用于有特殊需求的商家

#### 系统级别映射 (co_id = 2)
- 系统提供的通用映射规则
- 所有商家共享
- 由系统统一维护

#### 查询优先级
```
先查商家级别 → 未找到 → 再查系统级别
```

### 12.6 设计理念对比

#### 旧设计理念（支持商家级别）
- 允许商家自定义属性映射
- 灵活性高，可满足特殊需求
- 但增加了维护成本和数据不一致风险

#### 新设计理念（只查系统级别）
- 属性映射由系统统一管理
- 数据一致性好，维护成本低
- 性能优化（批量查询）
- 符合"系统自动计算，与用户操作无关"的理念

### 12.8 为什么只有类目保留商家级别支持？

| 原因 | 说明 |
|------|------|
| **业务差异大** | 不同商家对类目的理解和映射偏好差异较大 |
| **灵活性需求** | 商家可能需要根据自己的业务特点定制类目映射 |
| **标准化程度低** | 类目映射难以完全标准化 |

而属性/属性值/销售属性：
- 是平台标准化的数据
- 应该由系统统一管理
- 商家自定义会导致数据混乱

### 12.9 性能优化设计

#### 哈希索引

- 使用 `hash_id` 作为数据库索引
- 提升查询性能
- 使用 `mapping_key` 进行精确匹配，避免哈希冲突

#### 批量查询

```csharp
var dbMappings = GetAttributeMappingBatch(co_id, from_plf, to_plf, attr_ids);
```

- 一次查询多个映射
- 减少数据库往返次数
- 性能提升显著

1. **商品复制时** (主要来源) - 写入所有类型的映射
2. **商品保存时** (次要来源) - 仅写入类目映射,且使用商家级别

### 13.5 关键发现

#### 1. 商品复制是系统级别映射的主要来源

根据代码分析:
- ✅ 商品复制: 写入系统级别 (co_id=2),包含所有类型映射
- ⚠️ 商品保存: 写入商家级别 (co_id=实际商家ID),只有类目映射

#### 2. 商品保存的映射写入不是系统级别

之前文档中提到的"商品保存时写入系统级别映射"是不准确的,实际上:
- 商品保存时使用 `BuildCategoryKey(coid, ...)` (Line 4686)
- 这里的 `coid` 是 `CurrentUserContext.Coid` (当前商家ID)
- 不是系统级别的 co_id=2

#### 3. AddMappingToQueue 的"参数欺骗"设计

虽然 `AddMappingToQueue` 接收 `co_id` 参数,但内部强制使用 `SYSTEM_CO_ID=2`:

**代码位置**: `ShopItemCatMapService.cs` Line 646-700


这是一种"接口兼容性设计",保持方法签名不变,但内部强制使用系统级别。

### 13.6 总结

**商品复制时** (主要来源):
- 界面操作: 商品列表 → 复制 → 选择目标店铺 → 确认
- 写入时机: 每个店铺复制完成后立即写入
- 写入级别: 系统级别 (co_id=2)
- 写入内容: 类目+属性+属性值+销售属性 (全部)

**商品保存时** (次要来源):
- 界面操作: 商品编辑 → 保存/提交发布 (多个不同平台商品)
- 写入时机: 保存完成后写入
- 写入级别: 商家级别 (co_id=实际商家ID)
- 写入内容: 只有类目映射

**重要说明**:
- 商品复制是系统级别映射的主要来源
- 商品保存时写入的是"商家级别"类目映射,不是系统级别
- 两个时机的写入级别和内容都不同,不要混淆

## 十一、授权失效场景下的商品复制问题分析

> **问题发现时间**: 2026-03-04  
> **问题场景**: Shopee店铺(授权失效) 复制商品到 TikTok店铺(授权有效)  
> **问题现象**: 复制完成后,界面只显示类目,类目属性不显示;保存后再次编辑则正常显示

### 11.1 问题根源分析

#### 11.1.1 核心问题: session_status 字段传递错误

**问题链路**:

```
CopyNew方法(Line 4257) 
→ JSON深拷贝: JsonConvert.DeserializeObject(JsonConvert.SerializeObject(shopItem))
→ 源商品的session_status被复制到目标商品
→ 目标商品.session_status = "Invalid"(源店铺Shopee的状态)
→ 但实际应该是"Authorized"(目标店铺TikTok的状态)
```

**代码位置**: `ShopItemService.cs` Line 4221-4243 `CopyNew` 方法

```csharp
private ShopItem CopyNew(ShopItem shopItem, ShopLC shop)
{
    // 深拷贝源商品(使用JSON序列化/反序列化)
    var item = JsonConvert.DeserializeObject<ShopItem>(
        JsonConvert.SerializeObject(shopItem));
    
    // 🔴 问题: JSON深拷贝会把源商品的所有字段都复制过来,包括session_status
    // 源商品的session_status是源店铺(Shopee)的授权状态"Invalid"
    // 但目标商品应该使用目标店铺(TikTok)的授权状态"Authorized"
    
    // ... 其他字段重置 ...
    
    return item;
}
```

#### 11.1.2 前端判断逻辑

**代码位置**: `Standard.cshtml` Line 1047

```javascript
isAuthExpired() {
    return this.itemInfo.session_status && this.itemInfo.session_status !== 'Authorized'
}
```

**问题表现**:
- 复制后: `session_status = "Invalid"` → `isAuthExpired = true`
- 前端认为授权失效,跳过API调用
- 但`category_rules`已经在复制流程中正确生成并保存
- 问题是前端没有从`category_rules`恢复`rulesConfig`

#### 11.1.3 前端changeCategory方法的问题

**代码位置**: `Standard.cshtml` Line 1596

```javascript
if(this.isAuthExpired) {
    if(this.itemInfo.category_rules) {
        this.rulesConfig = this.itemInfo.category_rules
    }
    // 🔴 缺少: 初始化属性值的逻辑
    return
}
```

**问题**: 只恢复了`rulesConfig`,但没有初始化`product_properties`和`sales_properties`的显示,导致类目属性区域不显示。

#### 11.1.4 保存后为什么正常

**原因**: 保存时,后端`BuildShopItemTree`添加了正确的`session_status`(目标店铺的状态)

```
保存 → BuildShopItemTree → 使用目标店铺的shop.session_status
→ 再次加载时,session_status = "Authorized"
→ 前端isAuthExpired返回false,正常调用API
→ API使用目标店铺(TikTok有效)的Shop对象,调用成功
```

### 11.2 影响范围分析

#### 11.2.1 场景矩阵

| 场景 | 源店铺授权 | 目标店铺授权 | 当前表现 | 预期表现 |
|------|-----------|-------------|---------|---------|
| 1 | ✅ 有效 | ✅ 有效 | ✅ 正常 | ✅ 正常 |
| 2 | 🔴 失效 | ✅ 有效 | 🔴 类目属性不显示 | ✅ 应该正常显示 |
| 3 | ✅ 有效 | 🔴 失效 | ❓ 未测试 | ⚠️ 应该降级显示 |
| 4 | 🔴 失效 | 🔴 失效 | ❓ 未测试 | ⚠️ 应该降级显示 |

#### 11.2.2 复制返回数据分析

**实际返回数据**（来自测试）:

```json
{
  "si_id": 0,
  "shop_id": 899261256,
  "plf_id": "TikTok",
  "session_status": "Invalid",  // 🔴 错误: 这是源店铺Shopee的状态
  "category_rules": { ... },     // ✅ 正确: 已生成目标类目的规则
```

### 11.3 解决方案设计

#### 方案A: 后端修复session_status传递 ✅ 推荐

**修改点**: `CopyNew`方法中,复制后设置正确的`session_status`

**代码修改**:

```csharp

{
    var item = JsonConvert.DeserializeObject<G.Item.ShopItem.ShopItem>(
        JsonConvert.SerializeObject(shopItem));
    
    // ... 现有代码 ...
    
    // 🔴 新增: 设置目标店铺的授权状态
```

**优点**:
- 根本解决问题,数据源头正确
- 前端无需修改
- 代码复杂度不变

**缺点**:
- 需要传递shop对象到CopyNew方法

#### 方案B: 前端修复属性初始化

**修改点**: `changeCategory`方法中,授权失效时初始化属性显示

**优点**:
- 修改点单一
- 不影响后端逻辑

**缺点**:
- 治标不治本,`session_status`数据仍然错误
- 代码复杂度增加

#### 方案C: 前后端联合修复

**修改点**:
- 后端: 修复`session_status`传递(方案A)
- 前端: 增强授权失效时的降级逻辑(方案B)

**优点**:
- 最稳健,双重保障
- 覆盖所有边界场景

**缺点**:
- 修改点较多

### 11.4 推荐方案: 方案A

**理由**:
- 数据源头正确,符合"预防优于补救"原则
- 代码复杂度不变
- 前端逻辑已经正确(见《商品刊登授权失效页面优化设计》),无需修改
