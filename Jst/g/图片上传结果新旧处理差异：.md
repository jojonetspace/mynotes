
新旧逻辑差异对比
差异1：MainImg 的写入方式
旧逻辑（ReplaceMainImg）：

var img_list = media_info.ImgProperties.JsonClone();  // 先克隆 ImgProperties（强类型）
var mainimg = img_list.FirstOrDefault(x => x.url.Is(url));
mainimg.url = map.uploaded_url;
mainimg.upload_id = map.url_id;
// 再序列化回 img_list（Dictionary 格式）
media_info.img_list = JsonConvert.DeserializeObject<List<Dictionary<string, object>>>(img_list.ToJson());
新逻辑（ApplyUploadResultToShopItem）：

foreach (var img in shopItem.media_info.img_list)  // 直接操作 Dictionary
{
    var url = img.TryGetValue<string>("url");
    if (url.Is(downloadUrl))
    {
        img["upload_id"] = result.UploadId;
        if (result.UploadedUrl.IsNotNullOrEmpty()) img["url"] = result.UploadedUrl;
        break;
    }
}
差异：旧逻辑通过 ImgProperties（强类型）克隆后再序列化回 img_list，会保留 img_list 中所有原有字段（如 name、uid、width、height）。新逻辑直接操作 img_list 的 Dictionary，只修改 upload_id 和 url，其他字段不受影响。新逻辑更安全，不会丢失字段。

差异2：SkuVariantImg 的写入方式
旧逻辑：找到所有匹配 URL 的 SKU，克隆 img_property 后写入，再赋回。

新逻辑：同样克隆后写入，逻辑一致。✅

差异3：WebDetailImg 的处理
旧逻辑：替换 description 中的 URL，并追加 desc_url_mark。

新逻辑：同样处理，但有一个条件差异——旧逻辑在 InitMediaData 中已过滤掉 desc_url_mark 中已有的 URL（不重复上传），新逻辑在写回时检查 UploadedUrl 非空才替换。逻辑等价。

差异4：SizeChartImg 的写入
旧逻辑：通过 RepalceSizeChart 写入 size_chart.url 和 size_chart.upload_id。

新逻辑：直接写入，逻辑一致。✅

差异5：VideoFile 的处理
旧逻辑：写入 media_info.video_id。

新逻辑：同样写入 media_info.video_id。✅

差异6：时机不同（最关键）
旧逻辑：上传结果写入 ShopItemResourceUrlMapping 表（持久化），发布前再从表中查出来替换 shopItem（两步）。

新逻辑：上传完成后直接写回内存中的 shopItem（一步），发布时直接使用。

潜在问题：如果上传和发布在同一次 Task 执行中（ProcessResourceUpload → ProcessPublishTodoItem），内存中的 shopItem 是同一个对象，写回后发布时能直接用，没问题。

但如果 Task 重启（比如上传成功后服务器重启，下次 Task 再取到这条记录），todoItem.data 已被清空（上传成功后 data = ""），会进入 ProcessPublishTodoItem，此时从数据库重新加载的 shopItem 里 upload_id 是空的，发布会失败。

这是一个需要注意的问题：旧逻辑通过 mapping 表持久化解决了重启恢复问题，新逻辑没有持久化。

差异7：Lazada 视频的 cover_url
旧逻辑：ConvertMediaUploadResult 中会把视频封面图 URL 写入 map.cover_url，并持久化到 mapping 表。

新逻辑：ExtractUploadResult 中没有处理 cover_url，ApplyUploadResultToShopItem 中 VideoFile 分支也没有写 video_cover_url。

这是一个遗漏，Lazada 视频发布时可能需要封面图 URL。

总结
差异点	影响	建议
MainImg 写入方式	新逻辑更安全，不丢字段	✅ 无需修改
时机：内存 vs 持久化	Task 重启后 upload_id 丢失，发布失败	⚠️ 需要考虑
Lazada 视频封面图	cover_url 未写回 shopItem	⚠️ 需要补充
关于重启问题，有两个选项：

A. 接受这个限制——上传失败重试时会重新上传，重启后重新走一遍上传流程（data 为空 → ProcessPublishTodoItem → 重新计算图片 → 重新上传）

B. 在 ApplyUploadResults 后，把更新后的 shopItem.data 持久化回数据库
